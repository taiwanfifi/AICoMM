  背景：問題是什麼？

  現在大家想把 LLM 拆開跑——手機/邊緣裝置做前半段（prefill），雲端做後半段（生
  成答案）。問題是，前半段算完的「KV-cache」要傳到雲端，一個 14B 模型在 1024
  token 下就要傳 200MB。用 5G 上傳（100Mbps）要 16 秒，根本不能用。

  動機：別人怎麼做？為什麼不夠？
  方法: CacheGen
  做法: 把 KV 資料用 delta encoding + 算術編碼壓縮
  壓縮倍率: 3.5×
  問題: 還是在傳 KV 資料本身，壓不了多少
  ────────────────────────────────────────
  方法: H2O / SnapKV
  做法: 把不重要的 token 丟掉
  壓縮倍率: 2-4×
  問題: 單一模型內部用的，沒考慮跨模型、沒考慮無線網路會變
  ────────────────────────────────────────
  方法: INT4/INT8 量化
  做法: 把 KV 從 16-bit 壓到 4/8-bit
  壓縮倍率: 2-4×
  問題: 同上，而且 INT4 對某些模型會爛掉
  共同問題：全部都是「壓縮 KV
  資料再傳」，而且都是靜態的，不會根據網路狀況調整。

  我們的核心觀察

  同一個模型家族裡，小模型和大模型會關注同樣的位置。

  7B 覺得重要的 token 位置，跟 14B 覺得重要的位置有 83.7% 重疊。

  我們做了什麼？（Scout）

  既然位置都差不多，那就不要傳 KV 資料了，只傳「哪些位置重要」的編號就好。

  - 邊緣裝置用 7B 模型跑一遍，算出哪些位置重要

  先講 KV-cache 是什麼                                                          
                                                                                
  LLM 在讀你的輸入（context + question）時，每一層都會算出「這個 token 跟其他   
  token 的關係」，算完的結果存起來叫                                            
  KV-cache。之後生成答案時就不用重算，直接查這個 cache。

  KV-cache 就是模型的「閱讀筆記」——記錄了它讀完文章後對每個位置的理解。

  問題在哪

  邊緣裝置算完這份筆記後，要把它傳到雲端讓大模型接手。但這份筆記
  超級大（9.7MB），在無線網路上傳很慢。

  我們發現了什麼

  我們讓 7B（小模型）和
  14B（大模型）各自讀同一篇文章，然後比較：「你們覺得哪些位置重要？」

  結果：兩個模型挑出來的重要位置，有 83.7% 是一樣的。

  比如一段 100 個 token 的文章，保留 75 個最重要的：
  - 7B 挑了：位置 [3, 7, 12, 15, 22, 31, ...]
  - 14B 挑了：位置 [3, 7, 12, 15, 22, 33, ...]
  - 其中 63 個位置完全一樣（83.7%）

  所以我們的做法

  既然小模型已經知道大模型會看哪裡，那就：

  ❌ 原本：傳整份 KV-cache（9.7 MB 的浮點數矩陣）
  ✅ 現在：只傳一串數字 [3, 7, 12, 15, 22, 31, ...] （336 bytes）

  雲端收到這串編號後，自己跑一遍 prefill，但只關注這些位置，其他位置遮掉。

  336 bytes 怎麼算的

  大概 170 個 token 的文章，保留 75% ≈ 128 個位置。每個位置用 2 bytes（16-bit
  integer）存：

  128 × 2 = 256 bytes（加上一些 header ≈ 336 bytes）

  28,800× 是什麼意思

  9,700,000 bytes ÷ 336 bytes ≈ 28,800

  原本要傳 9.7MB，現在只傳 336 bytes，差了 28,800 倍。這就是壓縮倍率。

  CacheGen 壓縮完還要傳 2.8MB（3.5×），我們只傳 336 bytes（28,800×）。差了 8,000
   倍。

  代價是什麼

  雲端要 自己再跑一次 prefill（57ms）。但這比等網路傳
  9.7MB（780ms）快太多了，所以划算。

  那為什麼有人要傳 KV-cache？

  因為在大規模服務的場景下：

  1000 個使用者同時發 request
       ↓
  雲端 GPU 要幫每個人跑 prefill（很吃算力）
       ↓
  GPU 忙不過來，排隊等很久

  讓邊緣裝置先跑 prefill，把算好的 KV-cache 傳上去，雲端只要做 decode

  └─────────────┴────────────────────────────────────┴───────────────────┘
  Scout 的定位

  Scout 其實就是一個 聰明的折衷：

  - 雲端確實要自己重跑 prefill（57ms），跟直接傳原文一樣
  - 但 Scout 多給了一個資訊：「哪些位置重要」
  - 雲端只看那些位置，其他遮掉 → 效果等同於看完全文

  這在 bandwidth 很差的時候（10-50 Mbps）特別有用：傳 KV-cache 要等好幾秒，Scout
   幾乎零延遲。

  你說「context window 長一點不是更好？」

  Context window 長 → KV-cache 更大 → 更傳不動。所以 context 越長，Scout
  越划算：

  1K tokens:  KV = 9.7 MB,   Scout = 336 B    （28,800×）
  4K tokens:  KV = 38.8 MB,  Scout = 1.3 KB   （29,800×）
  32K tokens: KV = 310 MB,   Scout = 10.5 KB   （30,000×）

  Context 越長，Scout 的優勢越大。

  輸入：「台北是首都，人口260萬。請問首都人口？」（20 個 tokens）

  GPU 做的事：20 個 token 同時算，每個都跟其他 19 個比較關係
  計算量：O(N²)  ← N 是 token 數
  特性：一次算完，很吃 GPU 算力，但可以平行處理
  產出：KV-cache（每個 token 在每一層的 key 和 value 向量）

  Prefill 是算力密集型：GPU 核心越多越快。

  第二步：Decode（答題）

  一個字一個字生成答案。

  生成「2」→ 要回頭看 KV-cache 裡所有 20 個位置
  生成「6」→ 要回頭看 KV-cache 裡所有 21 個位置（多了「2」）
  生成「0」→ 要回頭看 KV-cache 裡所有 22 個位置
  生成「萬」→ 要回頭看 KV-cache 裡所有 23 個位置
  ...

  特性：每次只生一個字，但每個字都要讀整個 KV-cache
  瓶頸：不是算力，是記憶體頻寬（要反覆把 KV-cache 從記憶體搬出來）

  Decode 是記憶體頻寬密集型：VRAM 頻寬越大越快。

  所以兩步的瓶頸不一樣

  83.7% 怎麼算的？                                                             
                                                                               
  具體步驟：
                                                                               
  同一篇文章（~170 個 tokens），同一個問題
                                                                               
  7B 模型讀完 → 算 Q2C score → 挑最重要的 75%（~128 個位置）                   
  14B 模型讀完 → 算 Q2C score → 挑最重要的 75%（~128 個位置）                  
                                                                               
  比較兩個集合：
    7B 挑了：{2, 5, 8, 11, 15, 20, 23, 31, ...}  共 128 個
    14B 挑了：{2, 5, 8, 11, 15, 20, 25, 33, ...}  共 128 個
                                       ↑↑  這些不一樣

  重疊的有 107 個
  Overlap = 107 / 128 = 83.7%

  Q2C score 又是什麼？

  就是「query 有多關注這個位置」。

  文章：「台北 是 台灣 的 首都 ， 人口 約 260 萬」
  問題：「首都 人口 多少 ？」

  Q2C score = 問題裡每個字對文章每個位置的 attention 加總

  「台北」← 問題很關注它  → score 高
  「是」  ← 問題不太在意  → score 低
  「首都」← 問題超關注它  → score 最高
  「260」 ← 問題很關注它  → score 高
  「萬」  ← 問題很關注它  → score 高
  「的」  ← 問題不在意    → score 低

  Score 高的位置留下，低的遮掉。

  不 overlap 的 16.3% 是什麼？

  128 個位置裡：
    107 個：7B 和 14B 都挑了（overlap）
    21 個：7B 挑了但 14B 沒挑（7B 多選的）
    21 個：14B 會挑但 7B 沒給（漏掉的）

  14B 漏掉的那 21 個重要嗎？

  實驗結果：不重要。 p = 0.883，用 7B 的選擇 vs 14B
  自己的選擇，答案品質完全沒差。

  為什麼不重要？因為：

  - 那 21 個是 14B 排名 最後面 的（剛好在 75% 邊界附近）
  - 本來就不是很重要的位置，少了也沒差
  - 而 7B 多選的 21 個也不礙事——14B 的 attention 會自動忽略不重要的位置

  只用 overlap 的位置能生成好東西嗎？

  能。 直接看數據：

  14B 用自己挑的 128 個位置 → F1 = 0.664
  14B 用 7B 挑的 128 個位置  → F1 = 0.661

  差距 = 0.003，p = 0.883 → 統計上沒差別

  甚至在 50% 和 25% retention，7B 的選擇反而讓 14B 更好（+0.047,
  +0.059），雖然沒到顯著。

  Context length 是短的還是長的？

  主要實驗用短的，有額外測長的。
  ┌─────────────────┬───────────────────────────┬──────────────┐
  │      實驗       │       Context 長度        │ Overlap @75% │
  ├─────────────────┼───────────────────────────┼──────────────┤
  │ 主實驗（n=200） │ ~170 tokens（SQuAD 短文） │ 83.7%        │
  ├─────────────────┼───────────────────────────┼──────────────┤
  │ 長 context      │ 1024 tokens               │ 83.3%        │
  ├─────────────────┼───────────────────────────┼──────────────┤
  │ 長 context      │ 2048 tokens               │ 82.7%        │

  Overlap 從短到長幾乎不掉（83.7% → 82.7%），所以這個現象不是短文才有的。

  但 2048 是我們測的最長。 論文的 limitation 有承認沒測 4K 以上。14B +

  6. 怎麼「切開」拿到 KV-cache？

  不用切。HuggingFace Transformers 直接提供 API。

  # 跑一次 forward pass
  outputs = model(
      input_ids=tokens,
      output_attentions=True,       # ← 要求輸出 attention weights
      return_dict=True
  )

  # 直接拿到：
  kv_cache = outputs.past_key_values   # KV-cache（每層的 K 和 V 矩陣）
  attentions = outputs.attentions       # Attention weights（Q2C score
  從這裡算）

  # 算 Q2C score
  last_layer_attn = attentions[-1]           # 最後一層的 attention [1, heads,
  seq, seq]
  q2c_scores = last_layer_attn[:, :, n:, :n] # query→context 的部分
  scores = q2c_scores.sum(dim=(0,1,2))       # 加總 → 每個 context 位置一個
  score
  top_positions = scores.topk(k).indices     # 挑最高的 k 個位置

  沒有什麼 magic，就是正常跑 inference，然後從 output 裡拿 attention weights
   KV-cache。 Transformers 本來就支援，不需要改模型。

  唯一的限制：要用 attn_implementation="eager"（不能用 SDPA），因為 SDPA
  不支援輸出 attention weights。

### 🚨 盲點大檢驗：Reviewer 的無情炮火

#### 1. 【學弟的 Scout 模型 (KV-Cache 傳輸)】👉 **存在致命的邏輯盲點！**
*   **看似美好的 Story：** 手機端用 3B 模型算出「要看哪些字（Index）」，只傳 Index 給雲端的 14B 模型，省下傳 200MB KV-Cache 的頻寬。
*   **Reviewer 的致命攻擊（Fatal Flaw）：** 
    「等等，如果雲端只收到 Index，它怎麼憑空生出那些字的 KV 值？雲端還是得把原始的文字 Prompt 全部讀一次（重新跑一次 Prefill）啊！
    既然雲端反正都要重讀文字，**那為什麼手機不一開始就只傳『原始文字 Prompt』給雲端就好？** 一篇文字才幾個 KB，傳文字永遠比傳 KV-Cache 快！
    如果傳文字那麼快，雲端收到文字後，自己用 14B 模型做注意力挑選（SnapKV）就好了，**為什麼要浪費手機的電量去跑一個 3B 模型來教 14B 模型做事？**」
*   **結論：** 這篇論文的 Baseline（把 200MB 的 KV-Cache 透過 5G 傳輸）是一個**「稻草人（假想敵）」**。在現實中，沒人會透過 5G 傳 200MB 的 Cache，大家都是直接傳 Text。一旦被 Reviewer 抓到這個盲點，這篇論文會立刻被 Reject，老師會大發雷霆。

#### 2. 【SatFM 衛星基礎大模型】👉 **違反物理常識的盲點！**
*   **看似美好的 Story：** 用幾千萬筆衛星通道數據，訓練一個 Transformer 預測衛星通道。
*   **Reviewer 的致命攻擊：**
    「地面通訊因為充滿大樓反射（多徑效應），所以需要 AI 來學複雜規律。但是，**低軌衛星（LEO）到地面的通道是『視距傳播（Line-of-Sight, LoS）』為主！** 也就是說，衛星的衰減和都卜勒頻移（Doppler Shift）是完全可以透過幾何學、衛星軌道星曆和物理公式精準算出來的，計算複雜度是 $O(1)$。
    **你為什麼要用一個幾千萬參數的『黑盒子 AI』，去學一個早就被物理公式（Free-space path loss）完美解決的問題？這不是脫褲子放屁嗎？**」
*   **結論：** 如果只是單純預測通道，這在學術上是「用大砲打蚊子」。除非你能證明這個大模型能解決「物理公式算不出來的東西（例如極端天氣下的電離層閃爍、星際聯網的動態拓撲）」，否則很容易被老一輩的通訊專家打槍。

#### 3. 【Diffusion Models for PHY (實體層去噪)】👉 **忽略通訊本質的盲點！**
*   **看似美好的 Story：** Diffusion 準確率超高，把通道估測做得很完美。我們可以用 Consistency Model 加速它。
*   **Reviewer 的致命攻擊：**
    「在 6G 通訊中，通道的『相干時間（Coherence Time）』大約只有幾毫秒（ms）。也就是說，通道每幾毫秒就會變一次。
    你的 Diffusion 就算只跑 1 步，在 GPU 上的運算加上 I/O 延遲，算出來可能已經過了好幾毫秒了。**等你算出來那個超級完美的通道，現實中的通道早就已經變了！** 這個完美估測對真實通訊系統毫無意義。」
*   **結論：** 絕對不能把它用在「即時（Real-time）」的接收器上，這會被罵缺乏常識。只能把它包裝成「離線（Offline）的數位孿生建模」或「資料增強工具」。

#### 4. 【Agentic AI for Networks (多代理網路管家)】👉 **有道理，但存在安全隱患！**
*   **看似美好的 Story：** LLM 自己討論，自動去改網路的 Routing 表和資源分配。
*   **Reviewer 的致命攻擊：**
    「電信網路（如台哥大、中華電信）要求 99.999% 的可靠度。LLM 眾所周知會『幻覺（Hallucination）』。如果你放任 LLM 自己去改 5G Core 的參數，萬一它發瘋了，造成全台大斷網，誰負責？你這套系統怎麼證明它在數學上絕對安全（Safe and Verifiable）？」
*   **結論：** 概念極好，但如果論文裡沒有加上**「形式化驗證（Formal Verification）」**或**「安全沙盒（Safety Guardrails）」**，會被認為只是一個危險的玩具。
